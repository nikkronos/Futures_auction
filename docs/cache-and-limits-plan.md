# Damir — кэширование, лимиты и план доработок

**Статус:** 2026‑03‑07  
**Проект:** Виджет аукциона для T‑Инвестиции (Futures_auction)

## 1. Текущее состояние (прод)

**Сервер:** Timeweb, `/opt/futures_auction`, сервис `futures_auction.service`  
**URL:** `http://81.200.146.32:5000`  
**Доступ:** по корневому `/` показывается заглушка («Попросите свою личную ссылку»). Виджет только по `?profile=...` (nikita, damir, ivan и т.д.).

### 1.1 Текущий backend (server.py на проде)

- Версия с **серверным кэшированием v5** и правками марта 2026:
  - `GET /` без `profile` — заглушка (HTML без виджета и настроек).
  - `GET /?profile=...` — отдаёт `index.html` (виджет).
  - `GET /api/futures`, `GET /api/orderbook` — как раньше.
  - `GET /api/stats` — счётчик запросов, кэш, фоновый поток.
  - Фоновый поток обновляет до **40 инструментов**, интервал **4 сек** (аукцион) / 60 мин (обычное время).
  - Во время аукциона не кэшируется и не отдаётся из кэша стакан без цены аукциона.
  - Расчёт цены аукциона при одностороннем стакане (только bid или только ask).
  - Округление цен до 4 знаков в расчёте лотов (исправление «лоты 0» у фьючерсов).
  - Расписание аукционов без окна 19:00–19:05.

### 1.2 Текущий frontend (static/index.html на проде)

- Версия **v5**.
  - Профили через URL: `?profile=name`. Именные: ivan, damir, nikolay, alex, nikita.
  - Кнопки: **Обновить**, **Настройки**, **Авто: ВКЛ**, **+ Профиль**. Кнопки **«Ссылка»** нет.
  - При таймауте/ошибке запроса таблица не очищается — остаются последние данные + сообщение об ошибке.
  - Автообновление: **4 сек** в аукцион, **60 мин** вне аукциона. Лимит **40 инструментов**, счётчик в настройках.
  - Таблица: Актив, Цена 5м, Цена аукц., Откл. %, Лоты, Дисбаланс.

### 1.3 Репозиторий и прод

- Код в `Projects/Damir/` и в репозитории `nikkronos/Futures_auction` должен соответствовать проду после деплоя. Перед изменениями — бэкап на сервере (`cp server.py server_YYYY-MM-DD.py` и т.п.) и коммит.

## 2. Ограничения T‑Invest API (важно для всех решений)

По официальной документации `https://russianinvestments.github.io/investAPI/limits/`:

- Сервис котировок (MarketData / Quotes, REST/gRPC):
  - **600 unary‑запросов в минуту** на пользователя.
- Общий лимит по IP:
  - Рекомендуют не превышать **50 запросов/сек** (~3000/мин) и **1000 запросов/мин**.

Для нашего виджета:

- Одна полная «обновка» для одного инструмента по REST:
  - `GetOrderBook` (обязательно).
  - Плюс `GetCandles` (5‑минутная свеча) — для отклонения.
- При интервале **2 секунды**:
  - 30 обновлений/мин × N инструментов ≈ **30N запросов/мин на пользователя** только по стаканам.
  - С учётом свечей и нескольких пользователей лимиты легко приближаются к 600 запросов/мин на сервис котировок.
- Отсюда:
  - Без серверного кэша и ограничений, при большом числе инструментов × пользователей, виджет может упираться в лимиты или в время ответа API.

## 3. Наблюдаемые проблемы

1. Во время аукциона часть пользователей открывает **очень много тикеров** (десятки инструментов).
2. При включённом автообновлении (2 секунды) и нескольких таких пользователях:
   - На клиенте:
     - Красное сообщение: «Таймаут загрузки. Выберите меньше инструментов в настройках.»
     - Таблица может долго висеть в состоянии «Загрузка…».
   - На сервере / T‑Invest API:
     - Большое количество одновременных REST‑запросов, потенциальный выход к лимитам.
3. Попытка внедрения серверного кэширования v5:
   - Идея: сервер отслеживает активные инструменты, в фоновом потоке обновляет их и складывает в общий кэш; клиенты читают только из кэша.
   - Реализация (экспериментальная, в репозитории):
     - `MAX_CACHED_INSTRUMENTS = 18`; фоновый поток `_background_update_loop`.
     - Объединение активных инструментов по всем пользователям.
   - На практике:
     - На бою у части инструментов **перестали обновляться данные**, поэтому прод был откатан.

Вывод: сейчас виджет **в целом рабочий**, но:

- В пике аукциона (много тикеров + несколько пользователей) он чувствителен к перегрузке.
- Причина — отсутствие глобального ограничения на количество инструментов в запросе и отсутствие продуманного серверного кэширования.

## 4. Варианты решений (обсуждение)

### 4.1 Вариант A — Жёсткий лимит на клиенте (низкий риск, быстрый эффект)

**Идея:** полностью ограничить количество инструментов, которые реально уходят в запрос к `/api/orderbook`, на уровне фронтенда.

- Конкретно:
  - В `static/index.html`:
    - Ввести константу:  
      `const MAX_INSTRUMENTS_LIMIT = 20;`  
      `const MAX_INSTRUMENTS_PER_REQUEST = MAX_INSTRUMENTS_LIMIT;`
    - При формировании массива `ids` для запроса:
      - `const ids = Array.from(selectedIds).filter(Boolean).slice(0, MAX_INSTRUMENTS_PER_REQUEST);`
  - В UI настроек:
    - Показывать счётчик `выбрано/20`.
    - Если чекбоксов отмечено больше 20, показывать предупреждение:
      - «Показаны только первые 20 выбранных инструментов. Остальные не попадают в запрос (ограничение виджета).»

**Плюсы:**

- Не требуется трогать `server.py` и деплой бэкенда.
- Нагрузка на T‑Invest API становится предсказуемой:
  - Один пользователь → не более 20 инструментов × 30 обновлений/мин = 600 запросов/мин (на пределе, но без учёта свечей; в реальности чуть меньше).
  - При нескольких пользователях нагрузка растёт, но лимит на каждого пользователя по сервису котировок считается отдельно (на токен).
- Изменение локализовано в одном файле (`index.html`) → легко откатить, если что-то пойдёт не так.

**Минусы:**

- Пользователь, который выбрал, например, 40 инструментов, увидит только часть из них.
- Требуется понятный UX‑текст, чтобы не было ощущения «пропажи» тикеров.

### 4.2 Вариант B — Очередь запросов на сервере (средний риск)

**Идея:** на стороне `server.py` ограничить параллелизм запросов к `MarketDataService`.

- Реализация (в общих чертах):
  - Ввести глобальный «пул» или очередь запросов к T‑Invest:
    - Допускать, например, не более 1–2 одновременных HTTP‑запросов.
    - Между запросами ставить искусственную задержку (50–100 мс).
  - Все функции, которые обращаются к `GetOrderBook` / `GetCandles`, проходят через этот пул.

**Плюсы:**

- Даже при большом N и нескольких пользователях лимиты API контролируются.

**Минусы:**

- Чем больше инструментов и пользователей, тем больше задержка получения данных.
- Логика сервера усложняется; нужны тесты и мониторинг.

### 4.3 Вариант C — Серверное кэширование v5 (исправленная версия, средний/высокий риск)

**Идея:** вернуться к концепции v5, но перепроектировать и отладить до стабильного состояния.

- Основной подход:
  - Сервер хранит объединённый список **активных** инструментов (по всем пользователям).
  - Фоновый поток с заданным интервалом (2 сек в аукцион, 60 мин вне) опрашивает API:
    - Обновляет общие кэши для orderbook и свечей.
  - Все `GET /api/orderbook`:
    - Читают только из кэша.
    - Не ходят к API напрямую (или делают это только как fallback).
- Важные моменты для доработки:
  - Чёткое поведение, если инструмент **не попал в топ‑N активных** (`MAX_CACHED_INSTRUMENTS`):
    - Например: показывать «данных нет (лимит кэша)» и логировать.
  - Синхронизация доступа к кэшу и фоновому потоку.
  - Подробное логирование для отладки (по каждому инструменту).

**Плюсы:**

- Глобальная нагрузка на API зависит только от числа активных инструментов, а не от числа пользователей.

**Минусы:**

- Сложнее и рискованнее, чем простой клиентский лимит.
- Нужен тестовый период и возможность быстрого отката.

### 4.4 Вариант D — gRPC стримы (долгосрочный, отдельный проект)

**Идея:** перейти с REST‑polling на gRPC‑стримы котировок и стаканов.

- Возможности:
  - До 300 подписок на инструменты в одном стриме.
  - Отправка апдейтов примерно раз в 100 мс (по лимитам стримов).
  - Снимает часть ограничений по количеству REST‑запросов.

**Минусы:**

- Это уже не «маленький фикс», а полноценный архитектурный проект:
  - Нужен gRPC‑клиент на Python.
  - В идеале — WebSocket или другой механизм пуша данных на фронт.
  - Требуется отдельная спецификация, PoC и план миграции.

## 5. Рекомендуемый пошаговый план

### Шаг 0. Зафиксировать текущее состояние

1. Скопировать **актуальный** `server.py` с сервера в репозиторий как отдельный файл (например, `server_prod_2026-03-01.py`), чтобы не потерять боевую версию.
2. Убедиться, что в документах (этот файл, `ROADMAP_DAMIR.md`, `SESSION_SUMMARY_2026-02-28.md`, `DONE_LIST_DAMIR.md`) явно указано:
   - Бэкенд: версия без фонового кэша и без `/api/stats`.
   - Фронтенд: v5 с профилями, автообновлением 2 сек / 60 мин, `MAX_INSTRUMENTS_PER_REQUEST = 40` (пока).

### Шаг 1. Ввести жёсткий лимит на клиенте (рекомендуется первым)

**Цель:** защитить виджет от падения при «жадных» пользователях, не трогая сервер.

1. В `static/index.html`:
   - Установить разумный лимит, например:
     - `const MAX_INSTRUMENTS_LIMIT = 20;`
     - `const MAX_INSTRUMENTS_PER_REQUEST = MAX_INSTRUMENTS_LIMIT;`
   - При подготовке списка `ids` для `/api/orderbook`:
     - Использовать `slice(0, MAX_INSTRUMENTS_PER_REQUEST)`.
2. В блоке настроек:
   - Показывать: `выбрано/20`.
   - При превышении писать:
     - «Показаны только первые 20 выбранных инструментов. Остальные не попадают в запрос (ограничение виджета).»
3. Задеплоить **только** обновлённый `index.html` и протестировать на ближайшем аукционе:
   - С несколькими пользователями.
   - Отслеживать, исчезли ли массовые таймауты.

### Шаг 2. Добавить лёгкий `/api/stats` для диагностики (опционально)

1. В боевую версию `server.py` можно аккуратно добавить простой эндпоинт:
   - Счётчик запросов за последние 5 минут.
   - Может, без завязки на фоновый кэш (чтобы не повышать риск).
2. Использовать его только для мониторинга:
   - `curl localhost:5000/api/stats` во время аукциона.
3. Это облегчит анализ, если снова будут проблемы.

### Шаг 3. Перепроектировать и протестировать серверное кэширование v5 (если будет принято решение)

1. В отдельной ветке/копии:
   - Взять экспериментальную версию `server.py` с фоновым кэшем.
   - Добавить расширенное логирование по каждому инструменту:
     - Время последнего успешного обновления.
     - Ошибки обращения к API.
2. Развернуть тестовый стенд (можно с sandbox‑токеном).
3. Прогнать нагрузочный сценарий:
   - 20–30 инструментов.
   - Интервал 2 секунды.
   - Несколько «виртуальных» пользователей.
4. Только после успешных тестов — рассматривать деплой на прод.

### Шаг 4. Долгосрочно — рассмотреть gRPC стримы

1. В `Projects/Damir/docs/specs/` завести отдельную спецификацию:
   - Архитектура решения на gRPC.
   - Требования к серверу, мониторингу, логированию.
   - Стратегия миграции с REST на gRPC.
2. Реализовывать gRPC‑вариант как **отдельную** итерацию, не смешивая с текущими REST‑фикcами.

## 6. Краткое резюме для следующего агента

- Прод сейчас **рабочий, но хрупкий** в часы аукциона, если пользователи выбирают слишком много инструментов.
- Главный «быстрый» шаг для повышения стабильности:
  - Ввести жёсткий лимит на клиенте (20 инструментов) и честно сообщать об этом в UI.
- Серверное кэширование v5 и gRPC стримы — это следующие уровни улучшения, требующие отдельного проектирования и тестов.
